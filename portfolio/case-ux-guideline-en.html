UX Guideline Development

LULU.AI · Jan 2026 - Feb 2026

A project to turn vague UX language into executable criteria,
then connect it to an evaluation framework and production guidelines.

Summary

A guideline project that helps planners and designers evaluate UX with shared criteria.

Point 1
Vague statements such as “make it easy to use” needed to be translated into clearer and more measurable criteria.
Point 2
I created an evaluation framework to assess UX with more consistent judgment.
Point 3
I used AI to organize UX guidelines that could be referenced during actual production.
Point 4
I worked with a designer to define standards for aesthetics, an area that is harder for planners to judge alone.

Background

During UI design and collaboration, there was a clear need to define what each interface was for, which functions it included, and which variants needed to be handled. There was also no separate decision structure that teams could quickly reference in practice, so the documentation needed to support that as well.

Problem

When function and variant criteria were not clearly defined, even the same UI could be interpreted differently or designed in inconsistent ways. As a result, screen definitions accumulated without consistency, teams had to repeatedly revisit similar discussions, and UI choices still required unnecessary communication because there was no quick decision structure to rely on.

  • The same UI could be understood or designed differently depending on the person.
  • Screen-level feature definitions and compositions could become inconsistent.
  • Without a quick reference for decisions, repeated checks and communication were still required.

Goals

  • Create a document structure that can systematically organize UI features and variants through shared criteria.
  • Develop the document into a visually structured format that can be understood more quickly.
  • Provide faster decision support through decision trees and checklists.

Approach

1. I wrote a UI feature specification

I structured each feature by clarifying its purpose, included elements, and required data and types so the UI could be understood at the feature level.

2. I visualized the document with AI

I used AI to turn the text-heavy document into a more visual format so the information could be understood and shared more quickly.

3. I created decision trees and checklists

To support faster judgment during actual work, I organized decision trees and checklists that could reduce repeated discussion.

Outcome

  • Organized 14 UI features. (modals, GNB, and more)
  • Created one HTML document with AI so visualized materials and explanations could be reviewed together.
  • Compiled a set of decision trees and checklists.

What I Learned

I learned that the quality of a document depends less on how much information it contains and more on how quickly it can be understood and used for decision-making. When feature definitions, visualization, and judgment criteria were designed together, the document became a practical collaboration tool.
