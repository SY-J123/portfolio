# Self-Introduction Outline

마지막 업데이트: 2026-03-19

## 목적
이 문서는 자기소개서를 실제로 작성하기 전에, 각 문항에 어떤 내용을 넣어야 하는지와 그 이유를 정리한 설계 문서이다.

## 기본 원칙
- 문항은 `회사 관점에서 왜 뽑아야 하는가`가 보이도록 구성한다.
- 이직 사유는 길게 해명하지 않고, 짧고 긍정적으로 정리한다.
- 모든 문항은 직무와 연결되도록 쓴다.
- 추상 표현보다 실제 행동, 기준, 숫자를 우선한다.
- 각 문항은 한 줄 구조를 먼저 잡고, 그 구조 안에 사례를 배치한다.

## 변경 로그
- 기존 4문항 구조를 6문항 구조로 재편함.
- 문항 순서를 실제 자소서 작성 흐름에 맞게 아래와 같이 재정렬함.
  - 지원 동기
  - 경력 요약 / 직무 역량
  - 주요 성과 / 프로젝트 경험
  - 문제 해결 / 실패 경험
  - 협업 / 커뮤니케이션 경험
  - 입사 후 포부
- 기존 `AI를 활용한 협업 구조 개선` 사례는 정체성 문항이 아니라 `주요 성과 / 프로젝트 경험`과 `문제 해결 경험`에 배치하는 방향으로 정리함.

## 1. 지원 동기 (이직 사유 포함)

### 핵심 포인트
- 회사 기준으로 써야 한다.
- 이직 이유는 짧고 긍정적으로 정리한다.
- 지원 이유와 직무 적합성이 자연스럽게 이어져야 한다.

### 반드시 들어갈 내용
- 현재 역할에서 운영과 서비스 기획 경험을 통해 문제 정의와 구조 설계 역량을 쌓아 왔다는 점
- 현재 역할에서 얻은 경험은 크지만, 더 다양한 도메인과 더 넓은 문제를 다루고 싶다는 점
- 현재 서비스가 아직 출시 전이더라도, 이미 다음 단계로 확장하고 싶은 방향이 분명해졌다는 점
- 지원 회사의 제품, 서비스, JD를 보며 복잡한 요구를 구조화해야 하는 지점을 발견했다는 점
- 본인이 바로 기여할 수 있는 영역이 문제 정의, 협업 구조 정리, UX 흐름 설계라는 점

### 왜 이 내용이 들어가야 하는가
- 지원 동기는 호감 표현이 아니라 `왜 떠나고`, `왜 여기고`, `왜 내가 맞는가`를 짧게 증명하는 문항이기 때문이다.
- 이직 사유가 길어지면 방어적으로 보일 수 있으므로, 현재 경험의 한계보다 앞으로의 성장 방향을 중심으로 설명하는 것이 유리하다.
- 회사 선택 이유가 구체적이어야 채용 담당자가 `우리 회사를 제대로 보고 지원했는가`를 판단할 수 있다.
- 마지막에는 반드시 직무 적합성과 기여 가능성으로 연결되어야 한다.

### 한 줄 구조
`왜 떠나고 → 왜 여기고 → 그래서 내가 맞다`

### Open Item
- 지원 회사별로 `왜 여기인가`에 들어갈 제품, 조직, JD 포인트를 맞춤형으로 교체해야 함
- `아직 출시 전인데 왜 이직하느냐`에 대한 면접용 답변 논리는 별도 준비 필요

## 2. 경력 요약 / 직무 역량

### 핵심 포인트
- 첫 문장에서 본인을 어떻게 정의하는지가 중요하다.
- `쓸 수 있는 사람인가`를 빠르게 판단할 수 있어야 한다.
- 경력 설명은 역할 소개가 아니라 성과 기반으로 써야 한다.

### 반드시 들어갈 내용
- 본인을 한 줄로 정의하는 문장
  - 예: 운영 경험을 바탕으로 문제를 구조화하고 협업 가능한 형태로 바꾸는 서비스 기획자
- 경력 흐름
  - 운영 경험을 통해 서비스가 실제로 움직이는 방식을 이해하게 된 점
  - 현재는 서비스 기획자로서 기능 기획, 문서 구조 정리, UX 흐름 설계를 수행하고 있는 점
- 주요 업무 영역
  - 기능 기획
  - 문서 및 협업 구조 정리
  - UX 흐름 설계
- 핵심 강점
  - 문제의 본질을 다시 정의하는 능력
  - 협업 가능한 구조로 바꾸는 능력
  - 문서와 프로토타입을 상황에 맞게 설계하는 전달 능력
- 대표 성과는 숫자로 한 번은 제시
  - 예: 이해관계자 수 감소, 리드타임 단축, 페이지 코드 감소, boilerplate 절감

### 왜 이 내용이 들어가야 하는가
- 이 문항은 읽는 사람이 가장 먼저 `이 사람이 어떤 포지션의 사람인지` 판단하는 구간이기 때문이다.
- 도메인 자체보다 직무 역량이 먼저 보이도록 정리해야 새로운 산업으로 이동할 때도 설득력이 생긴다.
- 운영 경험과 기획 경험이 자연스럽게 연결되어야 현재 강점이 우연히 생긴 것이 아니라 축적된 결과로 보인다.

### 한 줄 구조
`나는 이런 사람 → 이런 일 함 → 이런 성과 → 그래서 잘함`

### Open Item
- 대표 성과를 어떤 숫자로 가져갈지 회사와 문항 길이에 따라 조정 필요
- 포지셔닝 문장을 더 짧고 강하게 다듬을 필요 있음

## 3. 주요 성과 / 프로젝트 경험

### 핵심 포인트
- 가장 중요한 문항이다.
- 반드시 STAR 구조로 쓴다.
- 결과는 숫자로 끝나야 한다.

### 반드시 들어갈 내용
- 상황
  - 문서 중심 협업 방식에서 커뮤니케이션 비용이 크고, 전달 기준이 모호했던 문제
- 목표
  - 단순 문서 작성 속도 개선이 아니라 협업 구조 자체를 더 명확하고 효율적으로 바꾸는 것
- 행동
  - 문제를 문서 작성 이슈가 아니라 협업 구조 이슈로 재정의함
  - PRD와 프로토타입의 역할을 나눠 전달 기준을 다시 설계함
  - AI 도구를 활용해 프로토타입 제작 속도를 높임
  - 공용 모듈화와 중복 제거를 통해 유지보수성과 토큰 사용 효율을 함께 관리함
  - 문서에는 요구사항과 기준을 남기고, 화면 이해가 필요한 부분은 인터랙션이 있는 프로토타입으로 보완함
- 결과
  - 이해관계자 투입: 3명 -> 1명
  - 업무 리드타임: 1~2주 -> 3~4일
  - 공용 컴포넌트: 14개
  - 프로토타입 페이지: 8개
  - 페이지당 평균 공유 모듈 참조: 9.3개
  - 코드 총량: 10,177줄 -> 9,579줄
  - 페이지 코드 감소: 약 29%
  - 신규 페이지 추가 시 boilerplate 절감: 70% 이상
- 임팩트
  - 문서 작성 노동보다 리서치, 인터랙션 검토, 플로우 설계, UX 판단에 더 많은 시간을 투입할 수 있게 됨
  - 소통 비용을 크게 줄이고, 기획 의도를 더 명확하게 전달할 수 있게 됨

### 왜 이 내용이 들어가야 하는가
- 이 사례는 단순히 AI를 써 본 경험이 아니라, 문제를 정의하고 구조를 설계하고 결과를 만든 대표 성과이기 때문이다.
- 실무자는 이 문항에서 `무엇을 바꿨고`, `내 역할이 무엇이었고`, `결과가 어느 정도였는지`를 가장 직접적으로 본다.
- 특히 이 사례의 핵심은 시간 단축이 아니라, 기획자가 더 본질적인 판단에 시간을 쓸 수 있게 구조를 바꿨다는 점이다.

### 한 줄 구조
`문제 → 내가 한 행동 → 성과`

### STAR 메모
- Situation: 문서 중심 협업의 한계와 높은 소통 비용
- Task: 전달 구조와 협업 기준 재설계
- Action: PRD/프로토타입 분리, AI 활용, 모듈화, 중복 제거
- Result: 리드타임 단축, 이해관계자 수 감소, 코드 효율 개선

### Open Item
- 메인 수치를 2개만 남길지, 3개까지 보여줄지 조정 필요
- 토큰 사용량 효율화는 보조 근거로만 활용

## 4. 문제 해결 / 실패 경험

### 핵심 포인트
- 실패 자체보다 회복 능력과 사고 방식을 보여주는 문항이다.
- 문제를 어떻게 분석하고 다시 개선했는지가 중요하다.

### 반드시 들어갈 내용
- 어떤 문제였는지
  - 처음에는 문서만 잘 쓰면 해결될 문제라고 봤지만, 실제로는 전달 구조와 협업 방식의 문제였다는 점
- 원인 분석
  - 문서 하나에 너무 많은 역할이 실려 있었고
  - 독자별 필요 정보와 전달 방식이 구분되지 않았으며
  - 시각화 없는 설명이 해석 차이를 만들었다는 점
- 해결 과정
  - 문제를 다시 정의하고 산출물의 역할을 분리함
  - 프로토타입을 활용해 이해 차이를 줄이고
  - 공용 구조를 만들어 반복 작업을 줄였음
- 결과
  - 리드타임과 이해관계자 수가 줄고, 전달 명확성이 높아짐
- 배운 점
  - 문제를 표면 증상으로 보지 않고 구조 문제로 다시 정의하는 것이 중요하다는 점
  - 도구보다 먼저 `무엇을 해결할 것인가`를 명확히 해야 한다는 점

### 왜 이 내용이 들어가야 하는가
- 실패 경험은 `실수했다`를 고백하는 문항이 아니라, 문제를 복기하고 더 나은 방식으로 전환한 사고력을 보여주는 문항이다.
- 같은 사례를 써도 성과 문항과는 다르게 `처음 판단의 한계`, `원인 분석`, `개선 논리`를 중심으로 재구성해야 한다.

### 한 줄 구조
`문제 → 원인 → 해결 → 개선`

### Open Item
- 실패 경험 문항이 별도로 있을 경우, 현재 사례를 `초기 접근의 한계` 중심으로 더 선명하게 다듬을 필요 있음
- 필요하면 다른 실패 사례를 별도로 발굴할 수도 있음

## 5. 협업 / 커뮤니케이션 경험

### 핵심 포인트
- `같이 일하기 괜찮은 사람인가`를 판단하는 문항이다.
- 갈등 상황에서 어떻게 기준을 맞추는지가 핵심이다.

### 반드시 들어갈 내용
- 협업 상황
  - 다양한 직군과 함께 일하면서 문서 범위와 완성 기준을 먼저 맞추는 것을 중요하게 생각한다는 점
- 갈등 또는 이슈
  - 조직마다 기획 문서가 다뤄야 하는 범위가 다르고
  - 비개발자는 최신 버전 확인 방식이나 문서 접근 방식에서 불편을 겪을 수 있다는 점
- 내가 한 행동
  - 문제의 범위와 업무의 범위를 먼저 정의하고 합의함
  - 독자가 실제로 읽고 활용할 수 있는 형식으로 문서를 설계함
  - 화면 이해가 필요한 부분은 프로토타입으로 보완함
  - 문서를 private 웹으로 배포하고, push 시 자동 업데이트되도록 만들어 접근성과 버전 확인을 쉽게 만듦
  - 의견이 다를 때는 감각적 표현보다 상대가 납득할 수 있는 기준으로 설명하려고 함
- 결과
  - 문서 접근성과 활용도가 높아지고
  - 전달 방식에 대한 혼선이 줄어들며
  - 협업 피로도가 낮아짐

### 왜 이 내용이 들어가야 하는가
- 기획자는 다양한 직군 사이에서 기준을 맞추는 역할이기 때문에, 협업 방식 자체가 핵심 역량으로 평가된다.
- 추상적인 성격 소개보다 `어떻게 조율했고`, `어떤 기준으로 설득했고`, `업무 방식을 어떻게 바꿨는지`가 훨씬 신뢰를 준다.

### 한 줄 구조
`갈등 → 조율 → 해결 → 결과`

### Open Item
- 실제 갈등 상황을 더 날카롭게 보여줄 한 개의 에피소드가 있으면 추가 검토
- `논리적으로 공감된다`는 피드백은 면접형 표현으로 전환 가능

## 6. 입사 후 포부

### 핵심 포인트
- 추상적으로 쓰면 안 된다.
- 회사와 직무에 연결된 성장 계획이어야 한다.

### 반드시 들어갈 내용
- 단기
  - 빠르게 조직과 제품을 이해하고, 현재 팀의 협업 방식과 문제 구조를 파악해 실무에 안정적으로 적응하겠다는 점
  - 문제 정의, 문서 구조화, UX 흐름 설계에서 빠르게 기여하겠다는 점
- 중기
  - 복잡한 요구를 구조화하고, 협업 비용을 줄이는 핵심 역할을 맡겠다는 점
  - 데이터와 사용자 경험을 함께 고려하는 기획자로서 판단 정확도를 높이겠다는 점
- 장기
  - 데이터와 UX를 함께 보는 서비스 기획자로 성장하겠다는 점
  - 회사 안에서 더 나은 의사결정 구조와 협업 문화를 만드는 데 기여하겠다는 점
- 성장 실행 근거
  - 정성적 판단의 한계를 느껴 데이터 기반 판단 역량을 강화하고 있다는 점
  - 방송통신대학교 학습과 통계 교재 독학을 통해 이를 실제로 보완하고 있다는 점
  - 출근 전과 점심시간에도 꾸준히 공부하고 있다는 점

### 왜 이 내용이 들어가야 하는가
- 입사 후 포부는 열심히 하겠다는 다짐이 아니라, 입사 후 어떤 순서로 적응하고 어떤 방식으로 성과를 내며 어떤 방향으로 성장할 것인지 보여주는 문항이기 때문이다.
- 단기, 중기, 장기로 나누면 추상적인 포부보다 실제 실행 계획처럼 읽힌다.
- 현재 이미 공부와 실행을 병행하고 있다는 근거가 있어야 선언적인 목표가 아니라 진행 중인 성장 방향으로 보인다.

### 한 줄 구조
`빠르게 적응 → 성과 → 성장 → 기여`

## 문항별 역할 요약
- 1번은 `왜 떠나고 왜 여기인지, 그래서 왜 내가 맞는지`를 설명한다.
- 2번은 `나는 어떤 경력과 직무 역량을 가진 사람인지`를 설명한다.
- 3번은 `나는 어떤 문제를 성과로 연결해 본 사람인지`를 설명한다.
- 4번은 `문제를 어떻게 복기하고 개선하는 사람인지`를 설명한다.
- 5번은 `나는 어떻게 협업하고 갈등을 조율하는지`를 설명한다.
- 6번은 `입사 후 어떤 순서로 적응하고 성장하며 기여할지`를 설명한다.

## 활용 메모
- 실제 작성 시에는 문항마다 모든 항목을 다 넣기보다, 핵심 3~4개만 골라 밀도 있게 쓰는 편이 좋다.
- 회사 맞춤형으로 바꿀 때는 1번과 6번을 가장 먼저 수정한다.
- 성과 문항은 반드시 숫자를 넣고, 문제 해결 문항은 반드시 원인 분석과 배운 점을 넣는다.
- 같은 사례를 여러 문항에서 재활용하더라도, 문항 목적에 맞게 강조점을 바꿔야 한다.
