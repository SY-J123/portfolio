# VOC 리뷰 분류 가이드

이 문서는 토스 앱 리뷰를 Maalej et al. (2016)의 4분류 체계에 따라 다중 라벨 분류할 때 사용하는 단일 기준이다. LLM 프롬프트는 이 문서를 참조해 구성되며, 분류 규칙이 변경될 경우 본 문서만 수정하면 전체 재분류가 동일한 기준으로 수행된다.

---

## 1. 분류 체계 (4유형)

한 리뷰에 최대 3개까지의 라벨을 허용한다. 해당되는 유형이 없으면 빈 배열(`[]`)로 반환한다.

### 1.1 버그 리포트 (`bug_report`)

**정의**: 앱의 오류·비정상 동작을 신고하는 리뷰.

**판별 신호**:
- 튕김·먹통·꺼짐·오류·에러 등 명시적 오류 언급
- 특정 기능이 작동하지 않는다는 서술 ("로그인 안 돼요", "송금이 실패해요")
- 예상 동작과 실제 동작의 불일치 ("버튼을 눌러도 반응이 없어요")

**예시**:
- "로그인하면 계속 튕겨요"
- "송금하려는데 오류 메시지가 뜨면서 안 됩니다"
- "푸시 알림이 며칠째 안 와요"

---

### 1.2 기능 요청 (`feature_request`)

**정의**: 새로운 기능의 추가나 기존 기능의 개선을 제안하는 리뷰.

**판별 신호**:
- "~해주세요", "~하면 좋겠어요", "~가 있으면" 등 요청·제안 어투
- 현재 없는 기능의 존재를 가정하는 서술 ("왜 X 기능은 없나요?")
- 특정 기능의 부재에 대한 아쉬움·불만 표현 ("다크모드 없어서 아쉽네요")
- 타 서비스와 비교한 개선 요구 ("카카오페이처럼 ~를 추가해주세요")

**예시**:
- "내역에 월별 필터 기능 추가해주세요"
- "다크 모드가 있으면 좋겠습니다"
- "결제 카드 우선순위 변경 기능이 필요합니다"
- "다크모드 없어서 아쉽네요" (특정 기능 부재 언급 → 기능 갭 신호)

---

### 1.3 사용자 경험 (`user_experience`)

**정의**: 사용 중 경험한 감정·불편·만족을 서술하는 리뷰. 구체적 오류나 기능 요청이 아닌 사용 경험에 대한 인상 서술.

**판별 신호**:
- 불편함·편리함·혼란스러움·헷갈림 등 감정·평가 서술
- 특정 동작·흐름에 대한 경험 묘사 ("송금 UI가 헷갈려요", "결제가 빨라서 좋아요")
- UI·UX의 이해도·직관성·만족도에 대한 평
- "처음 썼는데~" 같은 경험 자체에 대한 서술

**예시**:
- "송금 흐름이 헷갈려서 처음엔 어려웠어요"
- "UI가 깔끔하고 직관적이라 편합니다"
- "메뉴 구조가 너무 복잡해서 원하는 기능을 찾기 어렵습니다"

---

### 1.4 평가 (`rating`)

**정의**: 오류·요청·경험 언급 없이 단순 칭찬·비난·감탄만 담긴 리뷰.

**역할**: 이 카테고리는 실질 분석 대상이 아니며, 전처리에서 걸러지지 못한 **정보 가치가 낮은 리뷰를 모아두는 버킷** 역할이다. 주요 인사이트(버그 패턴·기능 요청·UX 이슈)는 나머지 3개 카테고리에서 도출하며, 대시보드의 범주별 주제 드릴다운(3.4)에서도 `rating`은 기본적으로 제외된다.

**판별 신호**:
- 구체적 내용 없이 평가 형용사만 사용
- 별점 기반 단순 소감
- 감탄사·욕설 단독

**예시**:
- "최고의 앱입니다"
- "별로예요"
- "그냥 쓸 만합니다"
- "쓰레기"

## 2. 다중 라벨 규칙

### 2.1 여러 유형이 섞인 경우
한 리뷰에 여러 내용이 담겨 있으면 해당되는 유형을 모두 붙인다.

**예시**:
- "앱이 자꾸 튕겨요. 그리고 내역 필터 기능 좀 추가해주세요" → `["bug_report", "feature_request"]`
- "UI가 헷갈리고 로그인도 자꾸 실패합니다" → `["user_experience", "bug_report"]`

### 2.2 라벨 개수 제한
- 최소 0개 (해당 유형 없음), 최대 3개
- 대부분은 1–2개로 분류됨

### 2.3 중첩 기준
- **버그 리포트 vs 사용자 경험**: 명시적 오류(튕김·먹통·실패)가 있으면 `bug_report`, 주관적 불편만 서술하면 `user_experience`
  - "송금이 안 돼요" → `bug_report`
  - "송금 흐름이 불편해요" → `user_experience`
- **기능 요청 vs 사용자 경험**: **특정 기능이 언급되었는지**로 판단한다. 기능명이 거론되면 어투와 관계없이(명시적 요청이든 부재에 대한 아쉬움이든) `feature_request`로 분류한다. 기능명 없이 일반적 감정·불편만 서술하면 `user_experience`
  - "다크모드 추가해주세요" → `feature_request`
  - "다크모드 없어서 아쉽네요" → `feature_request` (기능명 + 부재 불만 = 기능 갭 신호)
  - "뭔가 좀 아쉽네요" → `user_experience` (기능명 없음)
- **평가 vs 사용자 경험**: 구체성 기준. "좋아요"만 있으면 `rating`, "송금이 빨라서 좋아요" 처럼 이유·상황이 붙으면 `user_experience`

## 3. 경계 사례

### 3.1 모호한 경우
두 유형 중 애매하면 **본문에 더 구체적으로 서술된 쪽**을 우선한다.

### 3.2 부정적 감탄
"최악", "망했다", "다신 안 씀" 등 강한 부정 감탄만 있는 경우 → `rating` (구체 내용 없음)

### 3.3 고객센터·상담 언급
"고객센터 응답이 느려요" 같이 운영 차원의 불만은 서비스 경험이므로 → `user_experience`

### 3.4 질문만 있는 리뷰
"이거 어떻게 쓰나요?" 같은 단순 문의 → `user_experience` (앱 사용상의 혼란을 드러냄)

### 3.5 개발사 답변(reply) 처리
분류에 사용하지 않는다. 리뷰 본문(`text`)만 본다.

---

## 4. 입출력 포맷

### 4.1 입력
전처리된 리뷰 배열. 분류기에는 다음 필드만 전달한다:
```json
{
  "external_id": "...",
  "text": "..."
}
```

### 4.2 출력
각 리뷰에 `classification` 필드를 추가한다:
```json
{
  "source": "...",
  "external_id": "...",
  "author": "...",
  "score": ...,
  "text": "...",
  "posted_at": "...",
  "classification": {
    "types": ["bug_report", "feature_request"]
  }
}
```

- `types`는 `["bug_report", "feature_request", "user_experience", "rating"]` 중 해당되는 값의 배열
- 해당되는 유형이 없으면 빈 배열 `[]`

---

## 5. 배치 작업 규칙

- 입력 데이터: `data/*.preprocessed.json` 중 분석 기간에 해당하는 리뷰
- 출력 파일: `data/*.classified.json`
- 한 배치 크기: 20건 (너무 많으면 토큰 한도·품질 저하)
- 매 배치 시작 시 본 문서를 프롬프트에 포함
- 이미 분류된 `external_id`는 재분류하지 않음 (append 방식)
